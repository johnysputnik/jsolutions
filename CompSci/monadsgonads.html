Notes

Monads are a pattern that comes with the curse stated above.

Starting with functional programming there are two senses:

Programming with functions

Started with FORTRAN II in 1958.

Only becomes interesting when we have first class functions, higher order functions and lexical closure. Allowing passing of functions as arguments, setting as variables etc.

Pure Functional Programming

No mutation in functions, a more mathematical approach. Programs are easier to reason about. A function is a map from a set of values to a another set of values with no side effects.

Memoization and Caching are two examples of a different approach.

Remove assignments, loops and freeze all arrays and object literals.

This makes for a completely non practical language as in the real world everything changes. Immutability makes it hard to interact with the real world - no side effects such as IO for example.

The trick..

A function can take a function as an argument making each function being different.

Haskell has the IO Monad. A solution to a problem you should never have.

Check out Category Theory as the basis of monads. And learning Haskell helps.

It's easier to reason about Monads without the type system that Haskell has.

The following examples in JavaScript all return a monad: 

1: function unit(value)
2: function bind(monad, function(value))

Axioms: 

3: bind(unit(value), f) ==== f(value)
4: bind(monad, unit) ==== monad
5: bind(bind(monad, f), g) ====
6:     bind(monad, function (value) {
7:         return bind(f(value), g);
8:     })

This can have an OO transform: 

 9: bind(monad, func)
10: 
11: monad.bind(func)

Monads are a pattern that can be implemented as a macro in lisp, or as with functions in JavaScript: 

12: function MONAD() {
13:     return function unit(value){
14:         var monad = Object.create(null);
15:         monad.bind = function(func) {
16:             return func(value);
17:         }
18:         return monad;
19:     }
20: }

He also suggest an alternative context color that is based on nested levels.

An example of an identity monad: 

21: var identity = MONAD();
22: var monad = identity("Hello world.");
23: monad.bind(alert);

We can now call: 

24: monad.bind(f).bind(g)

This is similar to the ajax monad. And is similar also to the QString monad used in arg() call.

We now want to add methods to the monad: 

25: function MONAD() {
26:     var prototype = Object.create(null);
27:     function unit(value) {
28:         var monad = Object.create(prototype);
29:         monad.bind = function (func, args) {
30:             return func(value, ...args);
31:         }
32:         return monad;
33:     }
34: 
35:     unit.lift = function (name, func) {
36:         prototype[name] = function(...args) {
37:             return unit(this.bind(func, args));
38:         }
39:         return unit;
40:     };
41: 
42:     return unit;
43: }

Lift will take an ordinary function add it to the prototype and make it monad aware. An example of it's use: 

44: var ajax = MONAD().lift('alert', alert);
45: var monad = ajax("Hello world.");
46: 
47: monad.alert();

A maybe monad for handling null pointer exceptions: 

48: function MONAD(modifier) {
49:     var prototype = Object.create(null);
50:     function unit(value) {
51:         var monad = Object.create(prototype);
52:         monad.bind = function (func, args) {
53:             return func(value, ...args);
54:         };
55:         if(typeof modifier === 'function') {
56:             modifier(monad, value);
57:         }
58:         return monadl
59:     }
60:     return unit;
61: }
62: 
63: var maybe = MONAD(function (monad, value) {
64:     if(value === null || value === undefined) {
65:         monad.is_null = true;
66:         monad.bind = function() {
67:             return monad;
68:         }
69:     }
70: });
71: 
72: var monad = maybe(null);
73: monad.bind(alert);

Here we are changing the bind method to do nothing except return the monad. And we don't have to worry about null pointer exceptions.

Side note - Concurrency

Threading is bad with mutation. We then use mutexes which come with their own set of problems.

With purely functional programming we never have this problem but have no mutation.

Turn based processing - Never wait, never block, finish fast. Asynchronicity can be hard to manage. Used by node.js.

Nested event handlers are generally bad and promises are a much better solution.

Composition of promises are monads.

Side note - Actor Model

Forms the basis of scheme and solves a lot of our programming problems.

Futures and Promises as well as Monads generally come out of the Actor Model.
